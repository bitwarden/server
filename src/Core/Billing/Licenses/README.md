# Licensing system

## Overview

Bitwarden's **cloud** is the only party that can *issue* (create and sign) a license. A **self-hosted** instance can
only *verify* one. There is no license phone-home: a self-hosted server validates the license file it already holds,
offline, using embedded public certificates.

This single fact drives almost every design choice and constraint described below. Because the issuer (cloud) and the
verifier (self-host) run **independently-updated versions** of the server, the license format must tolerate version
skew in both directions — a newer cloud must issue licenses an older self-host can still validate, and a self-host's
existing license must keep validating after it upgrades.

> This document explains the **why** and the **constraints**. For the step-by-step **how-to** of adding a plan-gated
> organization ability (including the license step), see
> [Organization ability flags → Self-host licensing](../../AdminConsole/OrganizationFeatures/OrganizationAbility/README.md#8-self-host-licensing).

## Where licenses live

License files are stored on disk in the configured license directory:

- Organizations: `{LicenseDirectory}/organization/{orgId}.json`
- Users (premium): `{LicenseDirectory}/user/{userId}.json`

See `LicensingService.ReadOrganizationLicenseAsync`, `ReadUserLicense`, and `WriteUserLicenseAsync`.

## Two license formats coexist

Every license carries both a legacy and a modern proof of authenticity. Validation **branches on which is present**:
if `Token` is null or blank, the code falls back to the deprecated property-based path; otherwise it uses the
claims-based path.

|              | Legacy: property-based                                   | Modern: claims-based (JWT)                          |
|--------------|----------------------------------------------------------|-----------------------------------------------------|
| Proof        | RSA `Signature` over a serialized `name:value\|...` string | A signed JWT in the `Token` property                |
| Generated by | `OrganizationLicense` / `UserLicense` constructors       | `ILicenseClaimsFactory<T>` → `Create*TokenAsync`    |
| Validated by | `ObsoleteVerifyData` / `ObsoleteCanUse`                  | `VerifyData` / `CanUse` (reads a `ClaimsPrincipal`) |
| Status       | **Deprecated** — no new validation logic; see note below | **Preferred** — add new data here                   |

**When adding license-backed data, use the claims-based path.** The deprecated `OrganizationLicense` constructor
carries an in-code `⚠️ DEPRECATED` notice with the same guidance.

> "Deprecated" means **no new validation logic** goes in the legacy path — do not add comparisons to `ObsoleteVerifyData`
> or properties intended to be hashed. It does **not** mean the legacy path is inert: a new field still requires a
> one-line entry in the `GetDataBytes` ignore list (see [C2](#c2--legacy-signatures-are-version-gated-field-by-field))
> so it stays out of the legacy signature. That is the one unavoidable legacy touch when adding a field.

## Lifecycle

**Organization:**

1. **Create** — cloud builds claims (`OrganizationLicenseClaimsFactory.GenerateClaims`), then
   `LicensingService.CreateOrganizationTokenAsync` wraps them in a JWT (issuer `bitwarden`, audience
   `organization:{id}`, 1-year *token* lifetime — distinct from the license's own `Expires` claim).
2. **Store** — the license JSON is written to the self-hosted license directory.
3. **Validate** — `ValidateOrganizationsAsync` runs across enabled orgs (self-host only): reads the file (missing →
   disable), rejects a `LicenseKey` shared by multiple orgs, runs `VerifyData` against the live org record, and checks
   the signature for legacy licenses.
4. **Enforce** — on any failure, `DisableOrganizationAsync` sets `Enabled = false`, stamps an expiration, persists, and
   emails the billing contact.

**User (premium):** parallel but simpler. `ValidateUsersAsync` sweeps premium users; `ValidateUserPremiumAsync` is the
on-demand check with a **once-per-day** in-memory cache. Failure clears premium, emails the user, and pushes a
`PremiumStatusChanged` notification. `UserLicense`'s **legacy format is version 1 only** — it never grew the
property-version ladder `OrganizationLicense` has (the modern claims-based user path has no version gating at all).

## Certificate trust model

- Two known thumbprints exist: production and development.
- **Self-hosted** instances load the embedded `licensing.cer` (prod public key) to verify; non-production self-hosts
  also trust `licensing_dev.cer`. **Self-hosts cannot sign** — `SignLicense` throws when `SelfHosted`.
- **Cloud** loads a creation certificate (with private key) from a configured path, blob storage, or thumbprint.
- JWT validation (`GetClaimsPrincipalFromLicense`) enforces issuer, audience, signing key, lifetime, and **zero clock
  skew**. Token errors are deliberately re-thrown as `BadRequestException` so the client does not treat them as
  identity errors and force a logout.

## Constraints — why it is shaped this way

These are the load-bearing constraints. Modifying licensing without respecting them can **disable real customers'
organizations**.

### C1 — Backwards/forwards compatibility with self-hosted is the root constraint

The issuer (cloud) and verifier (self-host) run independently-updated versions. A license signed by a *newer* cloud
must still validate on an *older* self-host, and a self-host's license must keep validating after it upgrades. Every
constraint below is a consequence of this one.

### C2 — Legacy signatures are version-gated field by field

`OrganizationLicense.GetDataBytes()` reconstructs the exact signed string by including **only the properties that
existed at the license's `Version`** (e.g. `UsersGetPremium` is gated to ≥ v2, `UseScim` to ≥ v10). A new property
must be excluded from older versions' hashes, or previously-signed legacy licenses fail signature verification.

In practice, **new fields are not added as version gates at all** — they go in the trailing unconditional "ignore"
block (below the `// any new fields added need to be added here so that they're ignored` comment) so they never enter
the legacy hash. Fields such as `UseOrganizationDomains`, `UseRiskInsights`, and `UseMyItems` are excluded this way
regardless of version. The per-version gates above them exist only for fields that were hashed before this convention
settled.

### C3 — `CurrentLicenseFileVersion` is intentionally one behind

`CurrentLicenseFileVersion` is held one step behind the maximum version `ValidLicenseVersion` accepts — i.e. the
invariant `CurrentLicenseFileVersion == (max accepted version) - 1`. The concrete values live in
`OrganizationLicense.cs` (`CurrentLicenseFileVersion` and the `ValidLicenseVersion` range); this document deliberately
does not restate them so it cannot drift out of sync. Holding the current version one behind gives self-hosted
instances a grace window to upgrade before they receive "version not supported" errors. Bumping it is a deliberate,
coordinated act — not an automatic side effect of adding a field (note the `// TODO: Remember to change the constant`
reminder in the deprecated constructor).

### C4 — New claims must use a conditional `HasClaim` check (the PM-33980 trap)

This is the most dangerous gotcha. In the claims-based `VerifyData`, a self-hosted instance may hold a license file
generated **before** a given claim existed. `claimsPrincipal.GetValue<T>(...)` returns the type default (e.g. `false`)
for an absent claim, which then **mismatches** the live org value and disables the org. The fix (PM-33980) is to guard
every newly-added claim:

```csharp
(!claimsPrincipal.HasClaim(c => c.Type == nameof(UseMyItems))
    || useMyItems == organization.UseMyItems)
```

`UseMyItems` and `UseInviteLinks` are the current examples. **Every future claim added to `VerifyData` MUST follow this
pattern.**

### C5 — Validation is offline, time-boxed, and there is no revocation

A self-host validates locally; the cloud cannot revoke a license out-of-band. Control is exercised through **expiry +
grace period** (`LicenseExtensions` adds a grace period for annual subscriptions; monthly subscriptions get an annual
expiry to avoid forcing monthly re-uploads) and the **periodic validation sweep** that disables anything that no longer
matches. The user path adds a once-per-day check cache to bound validation cost.

### C6 — Adding a single field touches many coordinated places

A new license-backed field is not a one-line change. For the modern path it must be added to the **claims factory** and
to **`VerifyData`** (with the C4 guard). For continued legacy support it interacts with the **constructor**, the
**`GetDataBytes` ignore list**, and **`ObsoleteVerifyData`**'s version ladder. The duplication between the property
model and the claim model is the main source of manual, error-prone propagation when adding features.

> The how-to checklist for these edits lives in the ability guide's
> [Self-host licensing](../../AdminConsole/OrganizationFeatures/OrganizationAbility/README.md#8-self-host-licensing)
> section. This document explains *why* each edit is required.

## Known design tensions

Captured for awareness and for the licensing refactor effort
([PM-32220](https://bitwarden.atlassian.net/browse/PM-32220)); not prescriptive.

- **God-service.** `LicensingService` mixes certificate bootstrap, token creation, signing, file IO, and the org/user
  validation sweeps in one class.
- **Reflection-based hashing.** `GetDataBytes` builds the signed payload by reflecting over public properties and
  ordering by name. This is what makes the version-gating in C2 necessary and fragile.
- **Property ↔ claim duplication.** The same datum is expressed as a C# property (legacy) and a string claim (modern),
  validated in two places. New work pays the duplication tax (C6).
- **Two divergent date models.** Organization expiry uses subscription period + grace-period logic; user expiry is a
  flat `+7 days` bump. `UserLicense` also has two signing constructors (with / without subscription info), only ever at
  `Version == 1`.
- **`#nullable disable`.** `LicensingService`, `OrganizationLicense`, `UserLicense`, and both claims factories opt out
  of null-safety (each carries a `// FIXME: Update this file to be null safe` banner).
